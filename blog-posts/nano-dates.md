I trained a **1-million-parameter** model to turn natural date phrases into ISO-8601:

```
2024-03-10 | the 3rd of July 2025 => 2025-07-03
2024-03-10 | Jun 12 2023          => 2023-06-12
2024-03-10 | next week            => 2024-03-17
2024-03-10 | in 3 months          => 2024-06-10
```

It runs on a CPU in milliseconds, it was trained from scratch in about 30 seconds on
one GPU, and **every training example was generated by code** — no scraping, no
labelling, no distillation from a bigger model. The weights and a self-contained
inference file are on [Hugging Face](https://huggingface.co/vukrosic/nano-dates).

This post is about three things: why code-generated data is the right call for a
task like this, a bug that made the model look much smarter than it was, and the
clean line where a nano model's reasoning actually breaks.

## Why generate the data with code

A 1M-parameter model can't be a general assistant. But it can completely nail a task
that is **narrow and formally specified** — and date→ISO is exactly that.

The key move: the answer has a known structure, so you **sample the answer first**,
then render it in many natural surface forms. Pick a date, then emit it as
`June 12, 2023`, `Jun 12 2023`, `12 June 2023`, `the 12th of June 2023`, or — for
relative phrases — pick a reference date and emit `tomorrow`, `in 3 weeks`,
`next month`. Because you started from the answer, **the label is correct by
construction**.

That is strictly better than asking a large model to produce training data:

- **No verification.** The label isn't a guess you have to check — it *is* the
  ground truth you generated.
- **Free and unlimited.** 100,000 examples cost nothing and take a fraction of a
  second.
- **Total control of the distribution.** You decide exactly which forms, which date
  range, which difficulty.

Seventeen renderers, a byte-level tokenizer (vocab = 256, so any character is one
token, no vocab file), and prompt-masked supervised fine-tuning where only the
10-character ISO answer is scored — never the phrase the model was handed.

## The bug: a model that looked perfect by cheating

My first trained model scored **100% on every absolute date form**. "June 12, 2023",
"Jun 12 2023", "the 12th of June 2023" — all perfect. I almost shipped it as a date
*normalizer*.

Then I gave it a hand-typed example with a mismatch, and it failed instantly:

```
2024-03-10 | Jun 12 2023 => 2024-03-10   ← wrong! it echoed the reference date
```

The model wasn't parsing "Jun 12 2023" at all. It was **copying the reference date**
from the front of the prompt.

The cause was in my data generator. Every prompt starts with a reference date
(`today`) so the model can resolve relative phrases like "tomorrow". But for the
*absolute* forms, I had set that reference date **equal to the answer**. So during
training, the answer was always sitting right there at the start of the prompt — and
the model learned the obvious shortcut: ignore the phrase, copy the prefix. It scored
100% on a held-out set that had the same leak.

The fix is one line of intent: for absolute phrases, draw the reference date
**independently** of the answer. Now the answer is nowhere in the prompt except
encoded in the phrase itself, so the only way to be right is to actually parse it.

This is the whole danger of synthetic data in one example. Generated data is correct
by construction — but "correct" doesn't mean "not leaking the answer through a side
channel." **Your evaluation will happily confirm a model that learned your bug.** The
only thing that caught it was feeding the model an input the generator could never
produce.

## Where a 1M model actually breaks

After the fix, here's the honest capability map — held-out exact-match on 2,000 unseen
examples:

| capability | examples | accuracy |
|---|---|---|
| Parse absolute dates | `June 12, 2023`, `the 12th of June 2023`, … | **100%** |
| Simple relatives | tomorrow, yesterday, next/last week, next month, in N months | **98–100%** |
| Variable-N day/week math | in N days, N days ago, in N weeks | **77–81%** |
| Weekday resolution | next/last \<weekday\> | **~12%** |
| **Overall** | mixed | **85.4%** |

The interesting part is the bottom row. Everything the model can do, it does
reliably. The one thing it can't do is **"next friday."**

Why is that the hard one? Resolving a weekday phrase means mapping an arbitrary date
to its day-of-week and then doing modular-7 arithmetic to find the next matching day.
That day-of-week function is the most globally entangled computation in the whole
task — it depends on the full date, including century and leap-year structure — and a
1M-parameter model simply doesn't have the capacity to represent it. Adding training
didn't move it. That's not undertraining; it's a capacity ceiling, and it's a clean,
legible one.

So the honest framing of this model: it **parses absolute dates and resolves most
relative offsets**, and weekday arithmetic is where it stops. That's a far more useful
and trustworthy thing to publish than "100% accurate date parser."

## Try it

The [Hugging Face repo](https://huggingface.co/vukrosic/nano-dates) ships a
single-file model definition with no framework dependency — just `torch` and
`safetensors`:

```python
from modeling_nano_dates import load, parse

model = load("model.safetensors", "config.json")
print(parse(model, "2024-03-10", "the 3rd of July 2025"))  # -> 2025-07-03
print(parse(model, "2024-03-10", "in 3 weeks"))            # -> 2024-03-31
```
